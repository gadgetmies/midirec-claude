## Why

The just-archived `record-incoming-midi` change closed the recording half of the E2E loop тАФ notes played on a hardware controller now appear in the active channel's piano roll. The remaining half is playback: the app captures notes but has nothing to emit them with. This change adds an outbound scheduler that streams pre-emitted note-on/note-off messages to a real `MIDIOutput` with sub-millisecond hardware-level timing via the Web MIDI API's `timestamp` argument, so pressing **Play** turns recorded (or fixture) notes back into sound. Closing this loop is the prerequisite for every higher-level playback feature on the backlog (per-channel routing, looping playback, count-in, tempo automation), and finishes the third of the three core E2E slices announced when the runtime work began.

Single-output routing only in this slice тАФ every channel's notes go to the **first available** `MIDIOutput`. The Routing matrix is intentionally **not** consulted yet (per-channel output routing is its own backlog item, slice 6). Holding that surface flat keeps this change scoped to the scheduler, the panic semantics, and the transport-action contract.

## What Changes

- `useTransport().play()` and `useTransport().stop()` gain spec-level behavior contracts beyond timecode advancement. `play()` SHALL start the outbound scheduler from the current playhead; `stop()` SHALL halt the scheduler **and** emit panic (explicit note-off for every still-open note-on it dispatched, plus an `All Notes Off` CC on every channel that produced activity during the session). `pause()` is left out of scope тАФ it preserves timecode but its scheduler semantics are not specified by this slice (treated as `stop()`-equivalent for the scheduler in practice; the spec contract is documented only for `play()` and `stop()`).
- New module `src/midi/scheduler.ts` exposes a `MidiScheduler` (consumed via a top-level `useMidiScheduler()` hook mounted once in `App.tsx`, mirroring the recorder's lifecycle). Classic Chris Wilson "A Tale of Two Clocks" lookahead: a `requestAnimationFrame` loop ticks every frame; each tick walks every channel's `PianoRollTrack.notes`, finds notes whose start time (`note.t * msPerBeat`) falls within `[playheadMs, playheadMs + lookaheadMs)`, and pre-emits `MIDIOutput.send([0x90 | (chId тИТ 1), pitch, vel], timestamp)` for the note-on plus the matching `0x80 | (chId тИТ 1)` note-off at `(note.t + note.dur) * msPerBeat`, using `performance.now()`-anchored `timestamp` values so the Web MIDI implementation delivers them at sample-accurate hardware time.
- Lookahead window: **100 ms**. Absorbs ordinary frame skips while keeping per-frame work bounded. Notes whose `note.t * msPerBeat < playheadMs` are skipped (they belong to the past); notes whose start `>= playheadMs + lookaheadMs` are deferred to a future tick. A small per-channel cursor (last-scanned `t` index) prevents quadratic scanning across long takes.
- Output selection: read `useMidiOutputs().outputs[0]` at `play()` time. If `outputs.length === 0`, `play()` SHALL be a no-op AND emit a toast via `useToast().show('No output device available')`. The play button stays enabled (does not get disabled by missing output) тАФ the empty-handed click is observably a no-op with toast feedback, matching the existing transport ergonomics.
- Tempo capture is **once per session**: the scheduler snapshots `bpm` at the moment `play()` runs and uses that `msPerBeat` for the entire playback session. Live tempo changes during playback are out of scope (treated as documented "may break or no-op"). The same snapshot is used for both note-on and note-off `timestamp` arithmetic.
- Channel byte mapping: outgoing MIDI channel byte = `(channelId тИТ 1)` (our internal channel ids are 1-indexed `[1..16]`; the Web MIDI status byte's low nibble is 0-indexed `[0..15]`). The Routing matrix is **not** consulted in this slice.
- Mute/solo: the scheduler SHALL honor the existing `useChannels`-level mute/solo composition rules. Muted channels (or, when any channel is soloed, every non-soloed channel) SHALL have their notes skipped at scheduling time тАФ the channel produces no outbound MIDI for the duration of the playback session OR until the user toggles mute/solo (the scheduler re-evaluates on every tick, so a mid-playback un-mute resumes that channel's notes from the current `playheadMs` cursor, not from `t = 0`).
- Panic on `stop()`: the scheduler maintains an `activeNoteOns` map тАФ keyed by `(outputId, channelByte, pitch)` тАФ of every note-on the scheduler has dispatched but whose matching note-off has not yet been delivered (note-off timestamps are in the future relative to `stop()`). On `stop()`, the scheduler iterates the map and sends an explicit `[0x80 | channelByte, pitch, 0]` at `timestamp 0` (immediate) for each entry, then sends `[0xB0 | channelByte, 0x7B, 0x00]` (CC #123, All Notes Off, value 0) for every distinct `channelByte` that has produced any activity during the play session. The order is: note-offs first (precise), then All Notes Off (belt-and-suspenders for any synth that loses track). The map is cleared after panic.
- Disabled-state contract for the play button: unchanged in this slice. The button is **enabled** even when no output is available тАФ the no-op + toast path is the documented UX. (Recording's disabled-state contract is unchanged.)

## Capabilities

### New Capabilities

- `midi-playback`: live MIDI output scheduling тАФ the `useMidiScheduler` hook lifecycle, the lookahead scan and `MIDIOutput.send` dispatch, the per-session tempo snapshot, the mute/solo skip rules, the panic-on-stop semantics, and the `outputs[0]` selection contract. This is a new capability with no existing spec; `midi-runtime` owns *device enumeration* (and stops there), `transport-titlebar` owns the *button states*, `channels` owns the *data shape*. `midi-playback` is the glue that turns `PianoRollTrack.notes` into outbound `MIDIMessageEvent`s on a hardware output тАФ it has its own behaviors worth specifying separately (lookahead window, active-note map, panic ordering, mute/solo respect, single-output routing contract), exactly mirroring how `midi-recording` is its own capability for the inbound side.

### Modified Capabilities

- `transport-titlebar`: extend `useTransport().play()` and `useTransport().stop()` action contracts with the scheduler-driving behavior described above тАФ `play()` starts the scheduler with the current playhead and tempo snapshot; `play()` with no available output emits a toast and otherwise no-ops; `stop()` halts the scheduler and triggers panic before resetting `timecodeMs`. The existing rAF-driven `timecodeMs` advancement, the `playing`/`recording`/`mode` rendering rules, the play/pause icon toggle, and all visual contracts (LED, button accent, animations) are unchanged.

## Impact

- **Affected code**:
  - `src/midi/scheduler.ts` (new) тАФ `useMidiScheduler` hook with rAF loop, per-channel cursor, lookahead window, active-note map, panic emitter.
  - `src/hooks/useTransport.tsx` тАФ `play()` and `stop()` reducer branches gain side-effect hooks (callback registration) so the scheduler can subscribe to transitions. Implementation choice тАФ direct subscription vs. an effect inside `useMidiScheduler` watching `mode` тАФ lives in `design.md`.
  - `src/App.tsx` тАФ mount the scheduler hook once at the top level, alongside the existing recorder mount.
  - No changes to `src/components/transport-titlebar/Titlebar.tsx` (visuals unchanged); button states are unchanged.
- **Dependencies**: no new npm packages. Consumes `useMidiOutputs` (already exposed by `midi-runtime`), `useChannels` for the channel/notes data, `useTransport` for `mode`/`timecodeMs`/`bpm`, and `useToast` for the no-output message.
- **Out of scope** (each its own backlog item): per-channel output routing via Routing matrix, multi-output layering, output-channel rewrite, loop playback, count-in / pre-roll, tempo automation, CC and pitch-bend dispatch during playback, MTC/clock-source sync.
- **No spec deletions**. One existing capability picks up additive action contracts; one new capability lands.
- **Risk**: the rAF loop must not double-fire when the recorder's own rAF flush also runs in the same frame тАФ they are independent subscribers and should not contend. Decision noted in `design.md`. A second risk: the active-note map must be keyed by `(outputId, channelByte, pitch)` (not just `channelByte, pitch`) so a future multi-output routing slice can panic per output without revisiting this slice's data shape, even though only one output is used today.
